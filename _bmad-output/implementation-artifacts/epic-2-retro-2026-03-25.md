## Successes

### 1. Foundation-First Architecture
The protocol and pattern decisions in Story 2-1 (discriminated union types, `createWsMessage` factories, vanilla Zustand store pattern, server-authoritative model) became reusable templates that every subsequent story followed. By Story 2-5, most of the lifecycle logic was already built — the story was largely verification. This validates the architecture-first approach.

### 2. Cross-Platform Delivery
Every story shipped mobile and web simultaneously with consistent patterns. The `@jellysync/shared` package (protocol types, stores, constants) served as the contract between platforms, preventing drift.

### 3. Zero Blockers, Smooth Velocity
No blocking issues across all 5 stories. Minor TypeScript ecosystem paper cuts in Story 2-1 (tsconfig paths fix, useRef type annotation) were resolved quickly without derailing progress.

### 4. Testing Discipline
Test count grew steadily: 87 -> 105 -> 117 -> 100+ -> 141. Zero regressions across the entire epic. Co-located test pattern and comprehensive story dev notes ensured consistent test coverage expectations.

### 5. Story Intelligence Pattern
Each story file included detailed dev notes, established code patterns, previous story intelligence, and git intelligence sections. This context carried forward effectively, with each story building on the patterns documented in prior stories.

## Challenges

### 1. Hook Duplication Across Platforms
`use-websocket.ts` and `use-room.ts` were created separately for mobile and web with nearly identical logic. This creates maintenance burden and risk of drift. Platform-independent hooks should live in `@jellysync/shared`.

**Resolution:** Consolidate into shared package as part of Story 3-1.

### 2. Story Status Inconsistency
Stories 2-1 and 2-2 still show `status: review` in their markdown files while sprint-status.yaml marks them as `done`. This creates a misleading picture for anyone checking story files directly.

**Resolution:** Update story files to `done`; add status sync to completion checklist.

### 3. Pre-Existing Expo Template Error
The TypeScript error in `apps/mobile/index.ts` (Expo template `./App` import) has been carried across 4+ stories. Each story notes it as "pre-existing and unrelated" but never fixes it, eroding trust in build checks.

**Resolution:** Fix before Epic 3 kickoff.

### 4. TypeScript Ecosystem Paper Cuts (Story 2-1)
- Server tsconfig `paths` had to be removed because they mapped to shared source files outside `rootDir`
- `useRef<number>` needed explicit annotation to avoid const literal type inference with reconnect values
- Fastify websocket plugin must be awaited before `server.listen()` in tests

**Resolution:** Documented in story dev notes for future reference. Not systemic.

---

## Key Insights

1. **Foundation-first pays off** — Investing in protocol types, message factories, and store patterns in Story 2-1 accelerated every subsequent story. Story 2-5 being mostly verification proves the architecture was right.

2. **Consolidate shared logic** — Platform-specific components (React Native vs HTML) are fine to duplicate, but identical hooks across mobile/web should live in `@jellysync/shared`. The line is: if it has no platform-specific imports, it belongs in shared.

3. **Close the loop on process** — Story file statuses must stay in sync with sprint-status. Add to completion checklist.

4. **Clean up tech debt promptly** — Don't carry pre-existing errors across multiple stories. The cost of fixing is low; the cost of noise accumulation is higher.

---

## Team Agreements

- Component duplication across platforms is **acceptable by design** — don't try to unify React Native and web components
- Hook duplication where logic is identical is **not acceptable** — consolidate into shared package
- Story dev notes and intelligence sections are **valuable** — continue this practice in Epic 3
- Server-authoritative model is **working well** — maintain this discipline
