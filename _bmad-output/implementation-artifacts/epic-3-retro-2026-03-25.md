## Successes

### 1. TanStack Query Adoption — Foundation-First Pays Off Again
Brand new dependency adopted seamlessly in Story 3-1. The staleTime and cache config set up in 3-1 meant every subsequent story got instant cache hits when navigating between library and lobby. Zero additional cache work needed. Same pattern as Epic 2's protocol foundation accelerating all subsequent stories.

### 2. Organic UX Improvement
Story 3-3 discovered a better flow during implementation: "Create Room" navigates to the library first, and the room is created when the host picks a movie. This is a significantly better UX than creating an empty room first. Emerged naturally from the implementation, not from upfront design.

### 3. Architecture Enables Fearless Feature Work
Story 3-4 (mid-session movie swap) required zero server-side changes. The `handleMovieSelect` function was already fully implemented from Story 3-3. The server-authoritative model and clean architectural boundaries meant the entire story was purely client-side UI work.

### 4. Testing Growth and Infrastructure
Test count nearly doubled: 113 to 200. Story 3-4 established web app test infrastructure (vitest + @testing-library/react) — a lasting contribution that will benefit Epic 4. Zero regressions across all 4 stories.

### 5. Professional Frontend Polish
Glassmorphic design language is cohesive and premium-feeling: rim lighting on posters, shimmer loading states, swap confirmation bottom sheet, gradient buttons. Design token system ensures every new component looks consistent. Team committed to carrying this quality standard into Epic 4's sacred screen player UI.

## Challenges

### 1. Epic 2 Retro Action Items Not Followed Through
2 out of 4 action items from Epic 2's retrospective were not addressed, and the same problems repeated:
- Story file status sync was not added to completion checklist — 3/4 Epic 3 stories showed `status: review` instead of `done`
- Expo template TS error in `apps/mobile/index.ts` was not fixed — still carried across 9+ stories

### 2. Hook Duplication Repeated
Despite the Epic 2 agreement that identical hooks should live in `@jellysync/shared`, the `use-library` hook was duplicated across mobile and web with identical logic. The platform-specific import issue (`../../lib/auth`) continues to be the excuse for not consolidating.

### 3. Pre-Existing Errors Becoming Wallpaper
The Expo template TS error has been noted as "pre-existing and unrelated" in every story since Epic 1. Each story documents it but nobody fixes it. The error erodes trust in build checks and normalizes ignoring warnings.

### 4. Story File Status Drift
Same issue as Epic 2: sprint-status.yaml shows `done` but story markdown files show `review`. Fixed during this retrospective (6 stories updated: 2-1, 2-2, 2-5, 3-2, 3-3, 3-4).

## Key Insights

1. **Foundation-first continues to pay off** — TanStack Query in 3-1 accelerated all subsequent stories, just like the protocol patterns in 2-1 did for Epic 2. This validates the approach for Epic 4: Story 4-1 absorbs streaming research.

2. **Follow-through on retro commitments is our weakest area** — The same issues repeated across two retros. Soft commitments don't work. Hard rules with automation are needed.

3. **Architecture enables fearless feature work** — Mid-session swap required zero server changes because the server-authoritative model was already right. Clean boundaries pay dividends.

4. **Frontend polish is a strength to maintain** — The glassmorphic design language is cohesive and professional. This quality bar must be carried into Epic 4's sacred screen player UI.

5. **"Pre-existing, not my problem" is a debt accelerator** — Small errors compound when every story passes them through. One-epic grace period, then it becomes a blocker.

- Sub-500ms sync drift tolerance is technically demanding
- Jellyfin transcoding may introduce latency that complicates sync
- expo-video API surface is less mature than established video libraries
